mvc_05_differentUser_edit_delete

參考來源: mvc_05_differentUser_edit_delete

I updated the CSS styles but all the PHP remains the same from part 4. Here we cover several topics (Not in order): ACL, CRUD without AJAX, Path Fix, and a few minor things.

  • 新增users資料表欄位role，設定不同的使用者權限
  • 根據使用者登入的權限，來判斷是否要出現編輯使用者的超連結
  • 顯示所有使用者的資料
  • 新增使用者
  • 刪除使用者
  • 編輯使用者
  • 改變URL常數為遠端SERVER的網址

新增users資料表欄位role，設定不同的使用者權限

進入phpMyAdmin，在users資料表內，新增欄位role，並設定其資料型別為ENUM，和3個值為default、admin、owner。此role欄位的目地是，在使用者登入時，判斷其使用者的身份，來根據其使用權限來顯示不同資料。例如，只有管理者權限的才能修改其它使用者的其料，一般的使用者則不行。

改變使用者jesse的權限為owner，這樣jesse就能被允許進入管理使用者資料的介面

新增使用者joe，並設其權限為default，代表為一般使用者

根據使用者登入的權限，來判斷是否要出現編輯使用者的超連結

使用者登入時，login_model.php就會去資料庫讀取使用者的role欄值資料，來取得使用者的登入權限。你可運用echo data['role']來確定是否有成功取得權限的資料

確認有取得使用者的權限資料($data['role'])之後，新增Session變數role來代表使用者的權限資料

由於登入成功後，就會跳到controllers/dashboard.php，因此在這觀察$_SESSION的值是否有成功設定了

運行結果， 成功取得session變數，可看到取得了登入權限的變數role，和已登入的變數loggedIn

在之前的課程中，我們是利用Session的loggedIn變數的布林值來判斷是要出現login或是logout的超連結

現在在使用者登入成功後，我們要透過session的role變數，來判斷使用者的登入權限，以決定是否要顯示Users的超連結(Users的超連結是用來進入管理使用者的介面)

運行結果， 使用joe的帳號登入，由於joe的登入權限是一般使用者(defaults)，登入後，將不會看到管理員才能用的超連結。

一般的使用者登入後，不會出現管理人員使用的超連結。這樣就完成了運用session的role變數，來判斷使用者權限，顯示不同資料的功能了。

顯示所有使用者的資料

複製controllers/dashboard.php並命名為新的檔案controll/ers/user.php，因為這二個檔案的功能非常的相近，都是去處理資料庫的資料。Dashboard.php用來處理data資料表、user.php用來管理users資料表

修改controllers/user.php的內容，新增變數$role來取得使用者的登入權限，並運用render()來指定要顯示的頁面

在資料夾views\user之內新增檔案index.php，做為controllers/user.php所對應的要顯示的頁面

在views/user/index.php，寫入<h1>User</h1>

運行結果， 用管理員的身份登入後，點選超連結Users，就成功得跳到所對應的頁面，出現了<h1>標題User

之前的課程中，logout()用來跳到指定網址的寫法為
header('location: ../login')，這樣只能跳到網址的相對位置。現在改為
header('location: ' .URL. 'login')才能跳到網址的絕對位置

controllers/user.php要負責產生、編輯、刪去使用者資料等3種功能，因此分別新增create()、edit()、delete()等3個函式來分別處理這些功能

新增models/user_model.php做為之後要用來處理users資料表的sql運算用的檔案

在user_model.php中新增userList()，用來讀取users資料表的資料

新增變數userList，用來接收來自model->userList()所讀取出的users資料表的資料，以便將接收到的userList資料顯示至所對應的頁面上(views/user/index.php)

在views/user/index.php中輸出userList變數，觀察是否有成功接收到userList的變數資料

運行結果， 成功顯示出userList變數的資料，取得了來自users資料表的使用者資料了

新增使用者

成功取得users資料表的資料userList之後，運用foreach()將使用者的資料用表格的方示顯示。
並新增一表單，用來讓管理者新增使用者用，此表單會將表單欄位內的資料運用action的屬性，將資料傳至controllers/user.php內的函式create()來處理

為了確認controllers/user.php內的函式create()能否執行，寫入echo 1，看看是否會輸出數字1，以確認從表單新增使用者的功能是否能成功執行

運行結果， 點選表單的Submit按鈕，觀察是否能成功執行從表單新增使用者的功能

成功輸出字串1，這代表資料能成功傳到controllers/user.php的函式create()

在controllers/user.php的create()，運用$_POST來取得來自表單的資料，並用陣列變數$data來代表來自表單的資料。最後，將$data傳至相對應的model來處理新增使用者的sql指令

user_model.php接收到新增使用者的$data資料後，運用sql的insert指令，來將資料寫入users資料表中

運行結果， 在表單輸入新的使用者的資料後，就會成功的將資料寫入資料表內，並同時顯示在網頁上。

進入phpMyAdmin，在users資料表內可看到成功新增了新的使用者

將使用者資料成功運用model來寫入資料庫後，需運用header()將網址跳回user畫面的網址，不然會跳到user/create而呈現出空白的畫面

刪除使用者

刪除資料時，需要先取得要刪除資料的id值，因此需確認點選delete時，能正確的將對應的id資料傳送至controllers/user的delete($id)，因此超連結寫為<a href="'.URL.'user/delete/'.$value['id'] />

controllers/user.php的delete($id)，接受到要刪除資料的id值後，傳給相對應的model進行資料庫的處理，刪除完後，一樣運用header()來跳回user的畫面

user_model.php取得要刪除資料的id值後，運用sql的delete指令，將該筆id的資料從users資料表刪除

運行結果， 點選Delete，資料就可被刪去了，完成了刪除使用者資料的功能

編輯使用者

接下來， 我們來處理編輯使用者資料的部份。在controllers/user.php的edit($id)，將$id傳入對應的model函式userSingleList($id)，來取得此筆id的使用者資料

user_model.php的userSingleList($id)會運用sql的select指令來取得該筆id的資料

在views/user/index.php新增編輯用的超連結，點選該筆超連結時，會將id資料傳送至controllers/user.php的edit()，來執行編輯該筆使用者資料的動作

運行結果， 點選Edit的超連結，觀察是否會成功出現該筆使用者的資料

成功的出現所點選的使用者資料了。接下來， 就是製作編輯使用者資料的介面，來處理此筆id的使用者資料

在views/user/edit.php內新增編輯使用者資料的表單，將使用者的資料寫入表單的輸入欄位中，讓管理人員在編輯使用者資料時，可知道原有的資料是什麼，以便修改為新的資料。編輯完後，點選submit，會將資料傳至controllers/user.php的editSave($id)(注意:這部份原文的作者寫錯了，作者本來是想寫為user/editSave的，但是寫錯為userSave/edit，之後作者會發現並修改)

controllers/user.php需進行接收表單欄位的資料，作者認為這樣的修改並儲存表單資料的動作之前有在create()寫過相似的程式碼，因此將相似的程式碼貼入editSave($id)，一樣都是接收$_POST變數的表單欄位資料，只是修改對應的model函式為editSave($data)

運行結果， 出現錯誤，userSave not found

此時原文作者發現他表單的action屬性寫錯了，之前寫錯為userSave/edit，因該要寫為user/editSave，才能正確的將資料傳至controllers/user.php的editSave()

user_model.php的editSave($data)從controllers/user.php的editSave($id)接收到要編輯的使用者資料後，運用sql的UPDATE指令，來更新原有的資料表資料

由於controllers/user.php的editSave($id)需傳送此筆資料的id給相對應的model來進行運算，因此新增變數$data['id']來代表id值

運行結果， 點選第一筆資料的Edit超連結

將原本的login欄位資料從test2改為jesse，點選Submit觀察使用者資料是否會被成功修改

運行結果， 成功的將資用者的資料從test2改為jesse了，完成了編輯使用者資料的功能

改變URL常數為遠端SERVER的網址

如果你想將目前所完成的專案從測試用的網址移到正式的網址時，需要改變常數URL的值，例如作者是將常數URL從http://workspace/mvc改為http://jream.com/test/
